# Configuration

WyW-in-JS can be customized using a JavaScript, JSON or YAML file. This can be in form of:

- `wyw-in-js.config.mjs` ESM config file using `export default` (recommended for synchronous ESM config).
- `wyw-in-js.config.cjs` CommonJS compatibility config using `module.exports` (legacy mode).
- `wyw-in-js` property in a `package.json` file.
- `.wyw-in-jsrc` file with JSON or YAML syntax.
- `.wyw-in-jsrc.json`, `.wyw-in-jsrc.yaml`, `.wyw-in-jsrc.yml`, `.wyw-in-jsrc.mjs`, `.wyw-in-jsrc.js`, or `.wyw-in-jsrc.cjs` file.

Example `wyw-in-js.config.mjs`:

```js
export default {
  displayName: false,
};
```

`.mjs` config files are loaded synchronously. Keep them free of top-level `await` and async-only ESM dependency chains.

Legacy/CommonJS compatibility (`wyw-in-js.config.cjs`):

```js
module.exports = {
  displayName: false,
};
```

## Options

- `eval: EvalOptionsV2`:

  Controls v2 evaluation behavior (`strategy`, `runtime`, `errors`, `resolver`, `require`, `globals`, `onWarn`, and custom resolver/loader hooks).

  For `eval.strategy`:

  - `hybrid` (default): resolve values that can be proven static first, then fall back to the build-time evaluator.
  - `execute`: compute interpolation values with the build-time evaluator.
  - `static`: resolve only values that can be proven static. Values that would need evaluator fallback fail the build.

  `hybrid` is the recommended v2 default. It can use processor static semantics, statically resolvable imports, static
  metadata, and configured `staticBindings` without invoking the evaluator for those values, then fall back to the Node
  ESM evaluator for dynamic cases.

  Use `execute` as a compatibility escape hatch if a project depends on eval-time side effects or needs the closest
  behavior to always evaluating build-time values. Use `static` as a strict validation mode when any evaluator fallback
  should fail the build.

  `eval.runtime` selects the runtime used by the build-time evaluator. The current runtime is `nodejs`.

  `eval.errors` controls evaluator fallback errors:

  - `strict` (default): unsupported eval-time imports, non-literal `require()`, and non-string dynamic imports throw.
  - `loose`: unsupported cases emit warnings and use synthetic empty values where possible.

  For `eval.resolver`:

  - `bundler` (default): use the bundler resolver first, then native fallback when enabled by `eval.require`.
  - `native`: resolve eval imports with `oxc-resolver`. Native resolution discovers `tsconfig.json` by default, so TypeScript `baseUrl`/`paths` aliases are available unless `oxcOptions.resolver.tsconfig` overrides that behavior.
  - `hybrid`: try `eval.customResolver`, then native `oxc-resolver`, then the bundler resolver. This keeps package/relative imports and TypeScript aliases native-first while still allowing bundler aliases and virtual modules to resolve.
  - `custom`: use only `eval.customResolver`.

  For `eval.globals`:

  - Values are injected into the eval VM context for each transform run.
  - Changing a global value between runs is supported.
  - Removing a global key between runs is supported.
  - WyW invalidates eval cache/re-init state when the serialized `eval.globals` payload changes (not just keys, but values too).
  - Runtime mutations inside evaluated modules are not persisted as source of truth; the next run starts from `eval.globals` in config.
  - Supported value shapes are JSON-like primitives/arrays/plain objects plus functions and symbols; non-plain objects (for example `Date`, `Map`, `Set`, class instances) are rejected to avoid silent prototype/state loss.

  Keep `eval.globals` deterministic when possible (avoid `Date.now()`/`Math.random()` style values if you want stable cache behavior).

  Example:

  ```js
  export default {
    eval: {
      require: 'warn-and-run',
      globals: {
        BUILD_TARGET: 'web',
        FEATURE_FLAGS: { newNav: true },
      },
    },
  };
  ```

- `displayName: boolean` (default: `false`):

  Enabling this will add a display name to generated class names, e.g. `.Title_abcdef` instead of `.abcdef'. It is disabled by default to generate smaller CSS files.

- `variableNameConfig: "var" | "dashes" | "raw"` (default: `var`):

  Configures how the variable will be formatted in final CSS.

  Possible values

  - `var`

    Use full css variable structure. This is default behavior.

    ```js
    import { styled } from '@linaria/react';

    export const MyComponent = styled.div`
      color: ${(props) => props.color};
    `;
    ```

    In CSS you will see full variable declaration

    ```css
    .MyComponent_m1cus5as {
      color: var(--m1cus5as-0);
    }
    ```

  - `dashes`

    Removes `var()` around the variable name. It is useful when you want to control the variable on your own. For example you can set default value for CSS variable.

    ```js
    import { styled } from '@linaria/react';

    export const Theme = styled.div`
      --font-color: red;
    `;

    export const MyComponent = styled.div`
      // Returning empty string is mandatory
      // Otherwise you will have "undefined" in css variable value
      color: var(${(props) => props.color ?? ''}, var(--font-color));
    `;

    function App() {
      return (
        <Theme>
          <MyComponent>Text with red color</MyComponent>
          <MyComponent color="blue">Text with blue color</MyComponent>
        </Theme>
      );
    }
    ```

    In CSS you will see generated variable name and your default value.

    ```css
    .Theme_t1cus5as {
      --font-color: red;
    }

    .MyComponent_mc195ga {
      color: var(--mc195ga-0, var(--font-color));
    }
    ```

  - `raw`

    Use only variable name without dashes and `var()` wrapper.

    ```js
    import { styled } from '@linaria/react';

    export const MyComponent = styled.div`
      color: ${(props) => props.color};
    `;
    ```

    In CSS you will see just the variable name. This is not valid value for css property.

    ```css
    .MyComponent_mc195ga {
      color: mc195ga-0;
    }
    ```

    You will have to make it valid:

    ```js
    export const MyComponent = styled.div`
      color: var(--${(props) => props.color});
    `;
    ```

- `classNameSlug: string | ((hash: string, title: string, args: ClassNameSlugVars) => string)` (default: `default`):

  Using this will provide an interface to customize the output of the CSS class name. Example:

  ```js
  classNameSlug: '[title]',
  ```

  Would generate a class name such as `.header` instead of the default `.header_absdjfsdf` which includes a hash.

  You may also use a function to define the slug. The function will be evaluated at build time and must return a string:

  ```js
  classNameSlug: (hash, title) => `${hash}__${7 * 6}__${title}`,
  ```

  Would generate the class name `.absdjfsdf__42__header`.

  Last argument `args` is an object that contains following properties: `title`, `hash`, `index`, `dir`, `ext`, `file`, `name`. These properties
  are useful when you want to generate your own hash:

  ```js
  import sha1 from 'sha1';

  export default {
    classNameSlug: (hash, title, args) => sha1(`${args.name}-${title}`),
  };
  ```

  **note** invalid characters will be replaced with an underscore (`_`).

  Variables

  - `hash`: The hash of the content.
  - `title`: The name of the class.

- `variableNameSlug: string | ((context: IVariableContext) => string)` (default: `default`):

  Using this will provide an interface to customize the output of the CSS variable name. Example:

  ```js
  variableNameSlug: '[componentName]-[valueSlug]-[index]',
  ```

  Would generate a variable name such as `--Title-absdjfsdf-0` instead of the `@react/styled`'s default `--absdjfsdf-0`.

  You may also use a function to define the slug. The function will be evaluated at build time and must return a string:

  ```js
  variableNameSlug: (context) => `${context.valueSlug}__${context.componentName}__${context.precedingCss.match(/([\w-]+)\s*:\s*$/)[1]}`,
  ```

  Would generate the variable name `--absdjfsdf__Title__flex-direction`.

  **note** invalid characters will be replaced with an underscore (`_`).

  Variables

  - `componentName` - the displayName of the component.
  - `componentSlug` - the component slug.
  - `index` - the index of the css variable in the current component.
  - `precedingCss` - the preceding part of the css for the variable, i.e. `flex: 1; flex-direction: `.
  - `processor` - the processor used to process the tag (e.g. 'StyledProcessor' or 'AtomicStyledProcessor').
  - `source` - the string source of the css property value.
  - `unit` - the unit.
  - `valueSlug` - the value slug.

- `overrideContext: (context: Partial<vm.Context>, filename: string) => Partial<vm.Context>`

  A custom function to override the context used to evaluate modules. This can be used to add custom globals or override the default ones.

  ```js
  export default {
    overrideContext: (context, filename) => ({
      ...context,
      HighLevelAPI: () => "I'm a high level API",
    }),
  };
  ```

- <a id="importoverrides"></a> `importOverrides: Record<string, { mock: string } | { noShake: true } | { unknown: "allow" | "warn" | "error" }>`

  Overrides how imports are handled during prepare/eval stages.

  - Keys:

    - File imports (raw specifier starts with `.`) are keyed by a canonical root-relative path of the resolved file, e.g. `./src/foo.ts`.
    - Package imports are keyed by the raw specifier as written in code, e.g. `react` or `@scope/pkg`.
    - Keys support minimatch patterns. Exact keys take precedence; if multiple patterns match, the most specific one wins.

      **note** `?` is a minimatch wildcard. To match Vite query imports like `./icon.svg?react`, escape the question mark: `./icon.svg\\?react`.

  - Actions:

    - Each override value is one of:

      - `mock`: replace the resolved target for the import (raw `source` stays intact).
      - `noShake`: disable tree-shaking for this import (forces `only=['*']`).
      - `unknown`: controls behavior when an import reaches eval-time native resolver fallback (`warn` by default).

  Example:

  ```js
  export default {
    importOverrides: {
      './src/i18n/messages/en.json': { unknown: 'allow' },
      react: { mock: './src/__mocks__/react.ts' },
      './src/vendor/broken-module.ts': { noShake: true },
    },
  };
  ```

  Notes:

  - When WyW evaluates `__wywPreval`, it tree-shakes the module and removes `import '...';` side-effect imports by default, to avoid executing unrelated runtime code in Node.js (some libraries touch `document`, `window`, etc).
  - If your evaluated modules import browser-only packages (e.g. `msw/browser`), evaluation can still fail in Node.js. In that case, move the browser-only initialization out of evaluated modules, or mock the import via `importOverrides`.
  - If your evaluated modules contain runtime-only _heavy_ dependencies (e.g. code editors, syntax highlighters), build-time evaluation can become much slower even if those dependencies are imported lazily at runtime. In that case, mock the heavy import via `importOverrides` to keep evaluation fast.
  - If a module re-exports many values (e.g. a large “barrel” module like an icons index), WyW may need to re-run prepare for it multiple times as the required export set grows. If it shows up in slow-import warnings even after mocking the heavy leaf imports, add `{ noShake: true }` for the barrel module to force `only=['*']` and avoid incremental reprocessing.
  - If you need a side-effect import to run during evaluation, or you need to stub a problematic side-effect import, add an override for that import:
    - `{ noShake: true }` keeps the import and disables tree-shaking for that dependency.
    - `{ mock: './path/to/mock' }` keeps the import but redirects it to a mock module.

  Performance warnings (optional):

  - `WYW_WARN_DYNAMIC_IMPORTS=1` prints a warning when a module being prepared for evaluation contains `import()` that makes it to prepare stage. Even if it’s lazy at runtime (e.g. `React.lazy(() => import(...))`), WyW still needs to resolve/process it during build-time evaluation. If the target module is runtime-only or heavy, mock it via `importOverrides`.
  - `WYW_WARN_SLOW_IMPORTS=1` prints a warning when preparing a specific import takes “too long”. Tune the threshold via `WYW_WARN_SLOW_IMPORTS_MS` (default: `50`).

  Example:

  ```sh
  WYW_WARN_DYNAMIC_IMPORTS=1 \
  WYW_WARN_SLOW_IMPORTS=1 \
  WYW_WARN_SLOW_IMPORTS_MS=200 \
  <your build command>
  ```

  Example (stub a problematic side-effect import during evaluation):

  ```js
  export default {
    importOverrides: {
      '@radix-ui/react-tooltip': { mock: './src/__mocks__/radix-tooltip.ts' },
    },
  };
  ```

  Example (speed up evaluation by mocking a heavy runtime-only dependency):

  ```js
  export default {
    importOverrides: {
      '@uiw/react-codemirror': {
        mock: './src/__mocks__/uiw-react-codemirror.ts',
      },
    },
  };
  ```

  Example (mock a whole group of query imports, e.g. SVGR `?react`):

  ```js
  export default {
    importOverrides: {
      '@/icons/svg/*.svg\\?react': { mock: './src/__mocks__/svg-react.ts' },
    },
  };
  ```

- `outputMetadata: boolean` (default: `false`)

  Enables an opt-in metadata channel for downstream tooling.

  When enabled:

  - `@wyw-in-js/transform` includes normalized metadata in the transform result.
  - `@wyw-in-js/vite` emits `.wyw-in-js.json` sidecar assets during build.
  - `@wyw-in-js/cli` can emit matching sidecar files with `--output-metadata`.

  This does not change the default JS or CSS output when disabled.

  Example:

  ```js
  module.exports = {
    outputMetadata: true,
  };
  ```

- `importLoaders: Record<string, "raw" | "url" | ((context) => unknown) | false>`

  Allows handling of import resource queries (e.g. `?raw`, `?url`, `?svgUse`) during evaluation.

  Notes:

  - This affects only the WyW evaluation sandbox (values returned from `require()` / dynamic import during build), not your bundler runtime behavior.
  - If multiple query keys are present, the first matching key is used.
  - Set a loader to `false` to disable it.

  Built-in loaders:

  - `"raw"`: returns file contents as a string (`utf-8`).
  - `"url"`: returns a relative (POSIX) path to the resolved file from the importing file directory.

  Example:

  ```js
  export default {
    importLoaders: {
      // Disable defaults if you don't want them
      // raw: false,
      // url: false,

      // Custom loader key
      svgUse: (ctx) => ctx.toUrl(),
    },
  };
  ```

- `rules: EvalRule[]`

  The set of rules that defines how the matched files will be transformed during the evaluation.
  `EvalRule` is an object with two fields:

  - `test` is a regular expression or a function `(path: string) => boolean`;
  - `action` is an `Evaluator` function, `"ignore"` or a name of the module that exports `Evaluator` function as a default export.

  If `test` is omitted, the rule is applicable for all the files.

  The last matched rule is used for transformation. If the last matched action for a file is `"ignore"` the file will be evaluated as is, so that file must not contain any js code that cannot be executed in nodejs environment (it's usually true for any lib in `node_modules`).

  If you need to compile certain modules under `/node_modules/` (which can be the case in monorepo projects), it's recommended to do it on a module by module basis for faster transforms, e.g. `ignore: /node_modules[\/\\](?!some-module|other-module)/`. Example is using Regular Expressions negative lookahead.

  Information about `Evaluator`, its default setting, and custom implementations can be found in the [Transform Evaluator section](/how-it-works#transform-evaluator).

  The default setup is:

  ```js
  import { shaker } from '@wyw-in-js/transform';

  [
    {
      action: shaker,
    },
    {
      test: /[\\/]node_modules[\\/]/,
      action: 'ignore',
    },
    {
      test: (filename, code) => {
        if (!/[\\/]node_modules[\\/]/.test(filename)) {
          return false;
        }

        return /(?:^|\*\/|;|})\s*(?:export|import)[\s{]/m.test(code);
      },
      action: shaker,
    },
  ];
  ```

- `tagResolver: (source, tag, meta) => string`

  A custom function to use when resolving template tags.

  By default, linaria APIs like `css` and `styled` **must** be imported directly from the package so the static extractor can recognize the APIs. `tagResolver` allows `css` and `styled` APIs to be imported from other files too.

  `tagResolver` takes the source specifier from the import (eg. `@linaria/core` or `./my-local-folder/linaria`), the imported tag name (eg. `css`), and `meta`:

  - `meta.sourceFile` — the importer file path
  - `meta.resolvedSource` — best-effort resolved path for `source` (when available)

  It returns the full path to the related processor. If `tagResolver` returns `null`, the default tag processor will be used.

  For example, we can use this to map `@linaria/core` , `@linaria/react` or `@linaria/atomic` where we re-export the module.

  ```js
  import { createRequire } from 'node:module';
  import { dirname, join } from 'node:path';
  import { fileURLToPath } from 'node:url';

  const require = createRequire(import.meta.url);
  const __filename = fileURLToPath(import.meta.url);
  const __dirname = dirname(__filename);

  export default {
    tagResolver: (source, tag, { sourceFile, resolvedSource }) => {
      const pathToLocalFile = join(__dirname, './my-local-folder/linaria.js');
      const resolved =
        resolvedSource ??
        (sourceFile ? createRequire(sourceFile).resolve(source) : null);

      if (resolved === pathToLocalFile) {
        if (tag === 'css') {
          return require.resolve('@linaria/core/processors/css');
        }

        if (tag === 'styled') {
          return require.resolve('@linaria/react/processors/styled');
        }
      }

      return null;
    },
  };
  ```

  We can then re-export and use linaria API's from `./my-local-folder`:

  ```js
  // my-file.js
  import { css, styled } from './my-local-folder/linaria';

  export const MyComponent = styled.div`
    color: red;
  `;

  export default css`
    border: 1px solid black;
  `;
  ```

  ```js
  // ./my-local-folder/core.js
  export * from '@linaria/core';
  ```

- `oxcOptions: Object`

  Oxc-first parser, resolver, and transform options used by WyW when parsing, preparing, and evaluating modules.
  `oxcOptions.resolver` is also used by `eval.resolver: "native"` and the native step of `eval.resolver: "hybrid"`.
  By default, WyW passes `tsconfig: "auto"` to the native resolver. Bundler integrations may add static aliases from their
  own config; explicitly configured `oxcOptions.resolver.alias` entries take precedence.

  ```js
  export default {
    oxcOptions: {
      parser: {},
      resolver: {},
      transform: {
        define: {
          __DEV__: 'false',
        },
      },
    },
  };
  ```

- `features: Record<string, FeatureFlag>`

  A map of feature flags to enable/disable. See [Feature Flags](/feature-flags) for more information.

- `staticBindings: Record<string, Record<string, unknown>>`

  A per-source map of statically-known imported values used by `eval.strategy: "hybrid"` and `eval.strategy: "static"`.
  These bindings are one of the inputs to v2 static-first value resolution. Keys are either the import source as written,
  such as `@linaria/core`, or the resolved absolute file path. Values map imported names to either literal replacements
  or pure helper functions. Helper functions are called only when all call arguments can also be resolved statically.

  Use this for deterministic helpers or build-time constants that WyW cannot inspect directly. The trust model is the
  same as `importOverrides`: the project config vouches that these values are deterministic and intentionally match, or
  intentionally replace, runtime behavior.

  ```js
  export default {
    eval: {
      strategy: 'hybrid',
    },
    staticBindings: {
      '@linaria/core': {
        cx: (...args) => args.filter(Boolean).join(' '),
      },
      '/absolute/path/to/theme.ts': {
        themeVars: { panelBg: '#f00' },
      },
    },
  };
  ```

- <a id="coderemover"></a> `codeRemover: CodeRemoverOptions`

  An object with two keys `componentTypes` and `hocs`. It specifies types of variables and HOCs that will be replaced
  with an empty function before evaluation (useful to avoid executing runtime-only component/HOC code at build time).
  React `forwardRef` and `memo` are included in `hocs` by default.

  ```js
  {
    codeRemover: {
      componentTypes: {
        react: ['FC', 'FunctionComponent', 'ExoticComponent'],
        preact: ['FunctionComponent'],
      },
      hocs: {
        redux: ['connect'],
      },
    },
  }
  ```

## `@wyw-in-js/babel-preset`

The preset is a deprecated compatibility wrapper for environments that still invoke WyW through Babel. Bundler plugins use the Oxc-first transform path directly and do not depend on this preset internally.

Use this preset only when a non-bundler tool already runs Babel and you need WyW to participate in that pipeline, for example a legacy server build or a unit-test transform. Add `@wyw-in-js/babel-preset` at the end of the presets list:

`.babelrc`:

```diff
{
  "presets": [
    "@babel/preset-env",
    "@babel/preset-react",
+   "@wyw-in-js"
  ]
}
```

The preset understands the same option names as the configuration file, but not every option should be passed inline:

- The preset emits a compatibility warning once per process so it is harder to miss during migration.
- Inline preset options are forwarded to a separate Node process, so they must be JSON-serializable.
- Function-valued WyW options such as `tagResolver`, `eval.customResolver`, `eval.onWarn`, or custom `importLoaders` should live in a WyW config file instead.
- When you need those function-valued options, pass only `configFile` inline and let the preset load the rest from that file. That config file may be `.mjs`, but it still must be synchronous and must not use top-level `await`.

Example:

```json
{
  "presets": [
    "@babel/preset-env",
    "@babel/preset-react",
    ["@wyw-in-js", { "configFile": "./wyw-in-js.config.mjs" }]
  ]
}
```

If you only need simple inline options, keep them JSON-serializable:

```json
{
  "presets": [
    "@babel/preset-env",
    "@babel/preset-react",
    ["@wyw-in-js", { "displayName": true }]
  ]
}
```
